## 2. BVA Derivation & Rationale (Step-by-Step)

### Variable: `total_amount`
```
Type: INTEGER
No explicit app-level min/max. Logical business min = 1 (can't have zero-value order).
SQLite INT max = 2,147,483,647

Conservative boundary derivation:
  Min-1 = 0              → INVALID (zero-value order — logically invalid; behavior Unknown)
  Min   = 1              → VALID (minimum meaningful order amount)
  Min+1 = 2              → VALID

  Nominal = 150000        (typical order, e.g., 150,000 VND)

  Max-1 = 2,147,483,646  → VALID (within SQLite INT range)
  Max   = 2,147,483,647  → VALID (SQLite INT32 max)
  Max+1 = 2,147,483,648  → INVALID (SQLite INTEGER overflow — behavior Unknown)

Special cases:
  -1              → INVALID (negative — no business sense; behavior Unknown)
  -2147483648     → INVALID (INT32 min underflow)
  null            → INVALID (required field — behavior Unknown)
  missing field   → INVALID (behavior Unknown)
  "150000"        → Type mismatch (string instead of integer)
  1.5             → Type mismatch (float instead of integer)
  0               → Edge — zero amount, logically invalid (behavior Unknown)
```

> **Assumption A3 (restated):** Negative and zero values should be rejected by business logic but no validation code was found — tests will document actual behavior.

> **Assumption A7:** SQLite will silently truncate or error on values exceeding INT32 range. The exact behavior (crash vs. 500 vs. auto-truncate) must be verified.

### Variable: `shipping_address`
```
Type: TEXT (free string)
No format, min, or max length constraint found.

Conservative boundary derivation (min=1 char if required):
  Min-1 = ""           → Edge (empty string — required status Unknown)
  Min   = "A"          → VALID (1 char minimum)
  Min+1 = "AB"         → VALID (2 chars)

  Nominal = "123 Nguyen Hue, Quan 1, TPHCM"

  Max: No app limit → stress test at 500 chars and 5000 chars
  Max (stress) = "<500 x 'A'>"  → VALID (within TEXT)
  Overflow stress = "<5000 x 'A'>" → VALID (SQLite TEXT unlimited)

Special cases:
  null / missing     → behavior Unknown (required status not confirmed)
  spaces only "   " → Edge — meaningful address?
  Unicode / Vietnamese "Đường Lê Lợi, Quận 1" → VALID
  XSS payload        → Should sanitize
  SQL injection      → Should be blocked by parameterized query
  Emoji "🏠 123 St"  → System-dependent
```

### Variable: `Authorization` header (JWT Token)
```
Type: String (Bearer token)
Validation: middleware `authenticateToken`

Valid states:
  Valid token (active session)      → VALID → proceed to checkout
  
Invalid states:
  Missing header entirely           → INVALID → 401 Unauthorized
  "Bearer " with empty token        → INVALID → 401
  "Bearer invalid.jwt.token"        → INVALID → 401
  Expired token                     → INVALID → 401
  Token of deleted/non-existent user → INVALID → 401/403
  Token without "Bearer " prefix    → INVALID → 401
  Tampered token (modified payload) → INVALID → 401
  Token with role='admin'           → Behavior Unknown (should still create order as admin user)
```

### Variable: `user_id` (mass assignment attack surface)
```
Source: JWT token ONLY (req.user.id). Not from request body.

Attack vector: What if attacker sends `user_id` in request body?
  body: { user_id: 999, total_amount: 100, shipping_address: "X" }
  Expected: user_id from body is IGNORED; DB uses token-derived userId only
  Risk: If server uses body user_id → BOLA/IDOR vulnerability
```

### Variable: `status` (mass assignment attack surface)
```
System-hardcoded to "pending".

Attack vector: Attacker sends `status` in body:
  body: { total_amount: 100, shipping_address: "X", status: "completed" }
  Expected: status in body is IGNORED; DB always stores "pending"
  Risk: If server uses body status → business logic bypass
```
