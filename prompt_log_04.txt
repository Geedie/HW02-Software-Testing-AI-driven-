## 2. BVA Derivation & Rationale (Step-by-Step)

### Variable: `:id` (Path Parameter, Numeric BVA)
```
Type: INTEGER (SQLite AUTOINCREMENT)
Logical min = 1 (IDs start at 1)

Min-1 = 0        → INVALID (no order with id=0 exists by AUTOINCREMENT)
Min   = 1        → VALID if order #1 exists in DB
Min+1 = 2        → VALID if order #2 exists

Nominal = <seeded order ID from test setup>

Max = Undefined (SQLite INT max = 2,147,483,647)
  Max-1 = 2,147,483,646 → INVALID (no such order)
  Max   = 2,147,483,647 → INVALID (no such order)
  Max+1 = 2,147,483,648 → INVALID + potential overflow

Special:
  Non-existent valid integer (e.g., 99999) → 404 "Order not found"
  Negative: -1         → INVALID (no such order → 404 or 400)
  String: "abc"        → INVALID (type mismatch → 400 or 500)
  Float: "1.5"         → INVALID (type mismatch)
  SQL injection: "1 OR 1=1" → parameterized → should fail gracefully
  Path traversal: "../1"    → should be rejected by router
```

### Variable: `status` (EP1 Body, Enum + State Machine BVA)
```
Type: STRING ENUM
Valid set: {pending, confirmed, shipping, delivered, canceled}
Constraint: must also satisfy transition rule from currentStatus

Equivalence partitions:
  Valid enum + valid transition    → 200
  Valid enum + invalid transition  → 400 "Invalid state transition..."
  Invalid enum value               → behavior Unknown (no explicit enum check found)
  Empty string ""                  → behavior Unknown
  null / missing                   → behavior Unknown
  Case variant "Confirmed"         → behavior Unknown (likely invalid transition or ignored)
  Extra whitespace " confirmed "   → behavior Unknown

State Machine 2D Transition Matrix (currentStatus × targetStatus):
  [pending]   → confirmed: VALID ✓
  [pending]   → canceled:  VALID ✓
  [pending]   → shipping:  INVALID ✗
  [pending]   → delivered: INVALID ✗
  [pending]   → pending:   INVALID ✗ (self-transition — not in allowed map)

  [confirmed] → shipping:  VALID ✓
  [confirmed] → canceled:  VALID ✓
  [confirmed] → pending:   INVALID ✗
  [confirmed] → delivered: INVALID ✗
  [confirmed] → confirmed: INVALID ✗ (self-transition)

  [shipping]  → delivered: VALID ✓
  [shipping]  → canceled:  INVALID ✗
  [shipping]  → confirmed: INVALID ✗
  [shipping]  → pending:   INVALID ✗
  [shipping]  → shipping:  INVALID ✗ (self-transition)

  [delivered] → ANY:       INVALID ✗ (terminal state — no outgoing transitions)
  [delivered] → delivered: INVALID ✗ (self-transition)

  [canceled]  → delivered: VALID ✓ ← DEFECT-01 (anomalous but coded)
  [canceled]  → pending:   INVALID ✗
  [canceled]  → confirmed: INVALID ✗
  [canceled]  → shipping:  INVALID ✗
  [canceled]  → canceled:  INVALID ✗ (self-transition)
```

> **Assumption A7:** Self-transitions (e.g., `pending → pending`) are NOT in the allowed transitions map. The server likely returns `"Invalid state transition from pending to pending"`. Tests will confirm.

> **Assumption A8:** Status values not in the enum set (e.g., `"refunded"`, `"processing"`) are not explicitly handled — the server may return `"Invalid state transition from X to refunded"` or crash. Tests document actual behavior.

## 4. AI Gap Analysis & Vulnerability Report

### 4.1 Specification Gaps

| Gap ID | Description | Affected Variables | Risk Level | Recommendation |
|--------|-------------|-------------------|------------|----------------|
| GAP-001 | **DEFECT-01: `canceled → delivered` transition is allowed** — logically invalid; a canceled order cannot be delivered. This is a confirmed code bug. | `status` state machine | **CRITICAL** | Remove `canceled → delivered` from the allowed transitions map. Add test TC-OSM-030 to regression suite to detect if bug reintroduced. |
| GAP-002 | **DEFECT-02: User can cancel `shipping` orders** — developer comment explicitly flags this as unintended. Orders in transit should not be cancellable. | EP2 cancel logic | **CRITICAL** | Change cancel guard to: `if (order.status !== 'pending' && order.status !== 'confirmed') return 400` per developer comment. |
| GAP-003 | **No admin role check on `/api/admin/orders/:id/status`** — any authenticated user with a valid JWT may update any order's status. | `Authorization` / role | **CRITICAL** | Add: `if (req.user.role !== 'admin') return res.status(403).json({error: "Forbidden"})` before state machine logic. |
| GAP-004 | **`status` field not validated as required** — null/missing `status` may cause `undefined` to be passed into transition logic, leading to silent `"Invalid state transition from X to undefined"` or crash. | `status` body field | **HIGH** | Add: `if (!status) return res.status(400).json({error: "status is required"})` |
| GAP-005 | **No enum validation on `status` input** — values outside `{pending, confirmed, shipping, delivered, canceled}` (e.g., `"refunded"`, `"processing"`) are not explicitly rejected before transition logic. | `status` | **HIGH** | Add allowlist check: `const VALID_STATUSES = [...]; if (!VALID_STATUSES.includes(status)) return 400` |
| GAP-006 | **No case-insensitive handling or normalization** — `"Confirmed"`, `"SHIPPING"` silently fail without clear error message. Users/integrators may send mixed-case values. | `status` | **MEDIUM** | Normalize input: `status = status.toLowerCase()` before validation, OR document case-sensitivity explicitly and return clear enum error. |
| GAP-007 | **No whitespace trim on `status`** — `" confirmed"` or `"confirmed "` will fail transition logic silently. | `status` | **MEDIUM** | Add: `status = status.trim()` before processing. |
| GAP-008 | **No optimistic locking / version field** — concurrent status updates on same order have no conflict detection; last-write wins, creating race conditions in high-traffic scenarios. | `currentStatus` → `status` | **HIGH** | Add `version` INTEGER column to `orders`; require `version` in request body; reject if DB version ≠ request version (`409 Conflict`). |
| GAP-009 | **EP2 cancel: ownership validated by query filter only** — if SQLite `WHERE id=? AND user_id=?` returns 0 rows, it returns `"Order not found"` for both "wrong owner" and "doesn't exist" cases, hiding the true reason. | `user_id` ownership | **LOW** | Acceptable for security (don't reveal existence of other users' orders). Document explicitly. |
| GAP-010 | **`delivered` self-transition (`delivered → delivered`) behavior not explicitly handled** — server may return generic transition error or may silently succeed if transition map has a gap. | `status` state machine | **MEDIUM** | Add test TC-OSM-029 to confirm rejection; add explicit terminal state guard: `if (currentStatus === 'delivered') return 400` |
| GAP-011 | **No audit log / history for status transitions** — once an order's status changes, the prior state is lost. Forensic reconstruction of defective transitions (e.g., DEFECT-01) is impossible. | `status` | **MEDIUM** | Add `order_status_history` table: `(order_id, old_status, new_status, changed_by, changed_at)`. |
| GAP-012 | **`:id` path param type not validated** — string IDs like `"abc"` are passed directly to `SELECT status FROM orders WHERE id = ?`. SQLite may coerce or return no rows; error handling is implicit. | `:id` | **LOW** | Add: `if (isNaN(parseInt(id, 10))) return res.status(400).json({error: "Invalid order id"})` |
